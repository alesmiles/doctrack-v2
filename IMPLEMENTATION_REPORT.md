## Важные допущения (без остановки на вопрос, по «Режиму работы»)

**1. `<RightPanel>` из ТЗ = `src/components/base/SidePanel.tsx`.** Отдельного
компонента с именем `RightPanel` в репозитории нет; `SidePanel` — тот же
паттерн (правая выезжающая панель), уже используется в
`SoglasovanieDocumentsPage.tsx` для карточки документа. Использован как есть,
внутренняя логика не менялась.

**2. Значения `direction` — по существующему типу `DocDirection`, а не по
буквальному тексту ТЗ.** В коде уже был тип `DocDirection = "clients" |
"contractors_client" | "contractors_internal"` (soglasovanie-seed.ts), тогда
как ТЗ (R6) пишет `client` / `contractor-client` / `contractor-internal`.
Использованы реальные значения из кода, тексты и цвета бейджей — по R6
буквально.

**3. Дропдаун-фильтр «Клиент ▾ / КАМ ▾» со страницы Клиенты не вынесен в
переиспользуемый компонент** — в `ClientsPage.tsx` это инлайн-JSX
(`DropdownMenu` + `Button variant="outline"`), не отдельный компонент.
Переиспользован тот же визуальный паттерн и та же логика подсветки активного
состояния (`text-blue-700 border-blue-300 bg-blue-50`) один в один, но как
локальный код новой страницы, а не импорт общего компонента — переиспользовать
было нечего.

**4. `currentUser` был не связан с переключателем роли.** До правки
`App.tsx` всегда передавал `DEMO_USERS.find(u => u.role === "kam")` вне
зависимости от `currentRole` — значит переключение на «Директор» в футере
меняло видимые пункты сайдбара, но НЕ `currentUser.id`, и R2/R4
(«На подписи» видна/фильтруется по `currentUser`) физически не могли бы
сработать через сценарий Playwright-проверки («переключиться на роль
Директор»). Добавлен `ROLE_DEMO_USER_ID` и `currentUser` теперь выводится из
`currentRole`. Это правка существующего файла вне списка «не трогать» и
необходимое условие работы R1/R2/R4.

**5. `do_specialist` не имеет отдельного демо-пользователя в `DEMO_USERS`**
(там `kam/lawyer/producer/mendo/director/findir`) — в `ROLE_DEMO_USER_ID`
сопоставлен с `user-mendo` (ближайший по смыслу — менеджер ДО). Не влияет на
эту задачу (у `do_specialist` нет `canSign`), но нужен, чтобы `App.tsx` не
падал на `!` non-null assertion.

**6. `responsibleId` для документов «Клиенты»/«Подрядчики · клиенты».**
В исходных сид-данных `responsibleId` был проставлен только для
`contractors_internal` (для остальных направлений «Ответственный» не был
предусмотрен). Добавлен `responsibleId` документам `doc-c5`/`doc-cc4`/новым
документам, значения — правдоподобные сотрудники из `BASE_EMPLOYEES`
(точного кросс-маппинга КАМ-в-`RAW_PROJECTS` → сотрудник-в-`BASE_EMPLOYEES`
по имени в моках нет — это два независимых списка с разными демо-именами).

**7. Добавлены 2 новых документа в очередь директора.** В исходных
сид-данных `stage === "sign_us" && assignedTo === "user-director"` было
только 2 документа (`doc-c5` — «Клиенты», `doc-cc4` — «Подрядчики ·
клиенты»), направление «Подрядчики · внутр.» не было представлено вовсе, и
не было ни одного документа с `hasLawyerNote`. Для реальной проверки R6/R8/R9
(3 направления одновременно) и R16/R17 (замечание юриста, массовое действие)
добавлены `doc-c9` («Клиенты») и `doc-ci7` («Подрядчики · внутр.»,
`hasLawyerNote: true`). Итог: 4 документа в очереди директора вместо 2.

**8. Опции дропдаунов направления/Ответственный/Тип — из реальных
документов очереди, а не из полных `BASE_CLIENTS`/`BASE_CONTRACTORS`.**
Иначе дропдаун показывал бы контрагентов без единого документа в очереди —
бесполезный шум. Тот же принцип уже использован в
`SoglasovanieDocumentsPage.tsx` (`contragentOptions`).

**9. «Вернуть с вопросом» — заглушка** (`toast(...)`), логика возврата с
вопросом нигде в ТЗ не описана — R17 требует только наличие кнопки.

**10. Индикатор просрочки (красный текст + «+N дн.») в столбце «Подписать
до»** — не описан буквально в R5/R6, добавлен по аналогии с существующим
паттерном просрочки в канбан-карточке (`SoglasovanieKanbanCard.tsx`) и
`docUtils.ts`, для согласованности визуального языка.

**11. `SoglasovanieDocumentsPage.tsx` (канбан «Согласование · Документы») не
имеет пункта в сайдбаре** — это уже существовавший разрыв в навигации, не
связанный с этой задачей (страница рендерится только по прямому
`activePage === "soglasovanie-documents"`, никакой сайдбар-пункт на него не
ведёт). R18 «карточка уходит в „Подписан“ на доске Документы» проверено
структурно — обе страницы читают один и тот же стор
`useSoglasovanieDocuments()`, при `signDocuments()` обе получат новое
состояние синхронно — но не сквозным кликом через реальную навигацию, т.к.
такой навигации в интерфейсе сейчас не существует.

## Допущения

**1. Дропдаун «Подрядчик» технически содержит 3 пункта, а не 2** — 2
статусных значения («Новое», «В работе») + отдельное закрывающее действие
«Отправлено подрядчику» под разделителем (`DropdownMenuSeparator`). ТЗ
пишет: «Новое → В работе. При переходе в статус, эквивалентный "отправлено
подрядчику", строка удаляется» — при этом отдельно требует ровно 2
*значения статуса* в дропдауне и единый механизм смены через тот же
дропдаун. Совместить оба требования буквально (закрытие через тот же
дропдаун + ровно 2 статуса) можно только разделив «статус» (enum поля,
2 значения) и «закрывающее действие» (не значение статуса, а команда).
Дропдаун-список от этого содержит 3 пункта визуально, но `ContractorQueueStatus`
как тип имеет ровно 2 значения. DoD-проверка (отсутствие клиентских
статусов) проходит буквально.

**2. Видимость пункта «Согласование» не привязана к роли юриста.** ТЗ
называет раздел «рабочей очередью юриста», но нигде в DoD/Playwright-сценарии
не требует скрывать пункт для других ролей — в отличие от «На подписи»
(`currentUser.canSign`), где это было явным пунктом другого ТЗ. Чтобы не
домысливать интеграцию с `ROLES[...].sidebarItems` (не описанную в этом ТЗ),
пункт показан всем ролям.

**3. Счётчик в сайдбаре статичный** (`LEGAL_QUEUE_DOCUMENTS.length`), не
живой. Состояние списка (`documents`) живёт локально в `SoglasovanieQueuePage`
через `useState`, а не в общем сторе — в ТЗ нет требования синхронизировать
счётчик сайдбара с закрытием строк (в отличие от прецедента `useSoglasovanieDocuments`
в другой фиче, где общий стор был нужен явно для R18 другого ТЗ). DoD
буквально требует только равенства счётчика количеству строк «в текущих
тестовых данных» при загрузке — это выполнено.

**4. Счётчик «дней ожидания» для статуса «Отправлено, ждём клиента»**
реализован как отдельное поле `waitingSince` (дата отправки), а не выведен из
`deadline` — иначе колонка «Дедлайн» перестала бы означать одно и то же для
всех строк и ломала бы сортировку по дедлайну. Показан мелким текстом под
статус-бейджем.

**5. Формат дат в таблице** — `ДД.ММ.ГГГГ` (паттерн из reference-страницы
Клиенты), исходные данные в моке хранятся как ISO (`YYYY-MM-DD`) для
корректного сравнения при сортировке/просрочке.

**6. `TODAY` — общая константа `src/constants.ts` (`2026-06-12`)**, та же,
что используется для просрочки на странице Клиенты — не системная текущая
дата, чтобы демо-данные (просрочки, дни ожидания) оставались стабильными и
совпадали по смыслу с остальным приложением.
