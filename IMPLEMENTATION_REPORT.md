## Важные допущения (без остановки на вопрос, по «Режиму работы»)

**1. `<RightPanel>` из ТЗ = `src/components/base/SidePanel.tsx`.** Отдельного
компонента с именем `RightPanel` в репозитории нет; `SidePanel` — тот же
паттерн (правая выезжающая панель), уже используется в
`SoglasovanieDocumentsPage.tsx` для карточки документа. Использован как есть,
внутренняя логика не менялась.

**2. Значения `direction` — по существующему типу `DocDirection`, а не по
буквальному тексту ТЗ.** В коде уже был тип `DocDirection = "clients" |
"contractors_client" | "contractors_internal"` (soglasovanie-seed.ts), тогда
как ТЗ (R6) пишет `client` / `contractor-client` / `contractor-internal`.
Использованы реальные значения из кода, тексты и цвета бейджей — по R6
буквально.

**3. Дропдаун-фильтр «Клиент ▾ / КАМ ▾» со страницы Клиенты не вынесен в
переиспользуемый компонент** — в `ClientsPage.tsx` это инлайн-JSX
(`DropdownMenu` + `Button variant="outline"`), не отдельный компонент.
Переиспользован тот же визуальный паттерн и та же логика подсветки активного
состояния (`text-blue-700 border-blue-300 bg-blue-50`) один в один, но как
локальный код новой страницы, а не импорт общего компонента — переиспользовать
было нечего.

**4. `currentUser` был не связан с переключателем роли.** До правки
`App.tsx` всегда передавал `DEMO_USERS.find(u => u.role === "kam")` вне
зависимости от `currentRole` — значит переключение на «Директор» в футере
меняло видимые пункты сайдбара, но НЕ `currentUser.id`, и R2/R4
(«На подписи» видна/фильтруется по `currentUser`) физически не могли бы
сработать через сценарий Playwright-проверки («переключиться на роль
Директор»). Добавлен `ROLE_DEMO_USER_ID` и `currentUser` теперь выводится из
`currentRole`. Это правка существующего файла вне списка «не трогать» и
необходимое условие работы R1/R2/R4.

**5. `do_specialist` не имеет отдельного демо-пользователя в `DEMO_USERS`**
(там `kam/lawyer/producer/mendo/director/findir`) — в `ROLE_DEMO_USER_ID`
сопоставлен с `user-mendo` (ближайший по смыслу — менеджер ДО). Не влияет на
эту задачу (у `do_specialist` нет `canSign`), но нужен, чтобы `App.tsx` не
падал на `!` non-null assertion.

**6. `responsibleId` для документов «Клиенты»/«Подрядчики · клиенты».**
В исходных сид-данных `responsibleId` был проставлен только для
`contractors_internal` (для остальных направлений «Ответственный» не был
предусмотрен). Добавлен `responsibleId` документам `doc-c5`/`doc-cc4`/новым
документам, значения — правдоподобные сотрудники из `BASE_EMPLOYEES`
(точного кросс-маппинга КАМ-в-`RAW_PROJECTS` → сотрудник-в-`BASE_EMPLOYEES`
по имени в моках нет — это два независимых списка с разными демо-именами).

**7. Добавлены 2 новых документа в очередь директора.** В исходных
сид-данных `stage === "sign_us" && assignedTo === "user-director"` было
только 2 документа (`doc-c5` — «Клиенты», `doc-cc4` — «Подрядчики ·
клиенты»), направление «Подрядчики · внутр.» не было представлено вовсе, и
не было ни одного документа с `hasLawyerNote`. Для реальной проверки R6/R8/R9
(3 направления одновременно) и R16/R17 (замечание юриста, массовое действие)
добавлены `doc-c9` («Клиенты») и `doc-ci7` («Подрядчики · внутр.»,
`hasLawyerNote: true`). Итог: 4 документа в очереди директора вместо 2.

**8. Опции дропдаунов направления/Ответственный/Тип — из реальных
документов очереди, а не из полных `BASE_CLIENTS`/`BASE_CONTRACTORS`.**
Иначе дропдаун показывал бы контрагентов без единого документа в очереди —
бесполезный шум. Тот же принцип уже использован в
`SoglasovanieDocumentsPage.tsx` (`contragentOptions`).

**9. «Вернуть с вопросом» — заглушка** (`toast(...)`), логика возврата с
вопросом нигде в ТЗ не описана — R17 требует только наличие кнопки.

**10. Индикатор просрочки (красный текст + «+N дн.») в столбце «Подписать
до»** — не описан буквально в R5/R6, добавлен по аналогии с существующим
паттерном просрочки в канбан-карточке (`SoglasovanieKanbanCard.tsx`) и
`docUtils.ts`, для согласованности визуального языка.

**11. `SoglasovanieDocumentsPage.tsx` (канбан «Согласование · Документы») не
имеет пункта в сайдбаре** — это уже существовавший разрыв в навигации, не
связанный с этой задачей (страница рендерится только по прямому
`activePage === "soglasovanie-documents"`, никакой сайдбар-пункт на него не
ведёт). R18 «карточка уходит в „Подписан“ на доске Документы» проверено
структурно — обе страницы читают один и тот же стор
`useSoglasovanieDocuments()`, при `signDocuments()` обе получат новое
состояние синхронно — но не сквозным кликом через реальную навигацию, т.к.
такой навигации в интерфейсе сейчас не существует.

## Допущения

**1. Дропдаун «Подрядчик» технически содержит 3 пункта, а не 2** — 2
статусных значения («Новое», «В работе») + отдельное закрывающее действие
«Отправлено подрядчику» под разделителем (`DropdownMenuSeparator`). ТЗ
пишет: «Новое → В работе. При переходе в статус, эквивалентный "отправлено
подрядчику", строка удаляется» — при этом отдельно требует ровно 2
*значения статуса* в дропдауне и единый механизм смены через тот же
дропдаун. Совместить оба требования буквально (закрытие через тот же
дропдаун + ровно 2 статуса) можно только разделив «статус» (enum поля,
2 значения) и «закрывающее действие» (не значение статуса, а команда).
Дропдаун-список от этого содержит 3 пункта визуально, но `ContractorQueueStatus`
как тип имеет ровно 2 значения. DoD-проверка (отсутствие клиентских
статусов) проходит буквально.

**2. Видимость пункта «Согласование» не привязана к роли юриста.** ТЗ
называет раздел «рабочей очередью юриста», но нигде в DoD/Playwright-сценарии
не требует скрывать пункт для других ролей — в отличие от «На подписи»
(`currentUser.canSign`), где это было явным пунктом другого ТЗ. Чтобы не
домысливать интеграцию с `ROLES[...].sidebarItems` (не описанную в этом ТЗ),
пункт показан всем ролям.

**3. Счётчик в сайдбаре статичный** (`LEGAL_QUEUE_DOCUMENTS.length`), не
живой. Состояние списка (`documents`) живёт локально в `SoglasovanieQueuePage`
через `useState`, а не в общем сторе — в ТЗ нет требования синхронизировать
счётчик сайдбара с закрытием строк (в отличие от прецедента `useSoglasovanieDocuments`
в другой фиче, где общий стор был нужен явно для R18 другого ТЗ). DoD
буквально требует только равенства счётчика количеству строк «в текущих
тестовых данных» при загрузке — это выполнено.

**4. Счётчик «дней ожидания» для статуса «Отправлено, ждём клиента»**
реализован как отдельное поле `waitingSince` (дата отправки), а не выведен из
`deadline` — иначе колонка «Дедлайн» перестала бы означать одно и то же для
всех строк и ломала бы сортировку по дедлайну. Показан мелким текстом под
статус-бейджем.

**5. Формат дат в таблице** — `ДД.ММ.ГГГГ` (паттерн из reference-страницы
Клиенты), исходные данные в моке хранятся как ISO (`YYYY-MM-DD`) для
корректного сравнения при сортировке/просрочке.

**6. `TODAY` — общая константа `src/constants.ts` (`2026-06-12`)**, та же,
что используется для просрочки на странице Клиенты — не системная текущая
дата, чтобы демо-данные (просрочки, дни ожидания) оставались стабильными и
совпадали по смыслу с остальным приложением.

Опровергнута: `UploadZone.tsx:69-82` — `onClick={() =>

нативного `.click()` в `page.evaluate`) — счётчик реального React `onClick`
всё равно показал **2**. Это доказывает, что двойное срабатывание — не
артефакт тестового инструмента, а воспроизводимый баг на уровне кода.

Факт: **НЕТ.** Инструментировал `onOpenChange` в `CreateClientDocModal.tsx`

предположительно. (Также подтверждено визуально: `ref` DOM-узла `dialog`
в снапшотах до и после клика — тот же экземпляр, поля формы, заполненные
до клика — «маркер-теста-D17» в комментарии — сохранились после.)

**Пункт 4 — обёрнут ли `fileRef.current?.click()` в асинхронный код
(`setTimeout`/`Promise`/`await`)?**
Факт: **НЕТ.** Прямое чтение кода: `onClick={() => fileRef.current?.click()}`
— синхронная стрелочная функция без единого `await`/`setTimeout`/`.then`.
Подтверждено и во время диагностики через `Date.now()`-метки в
`console.log` до и после вызова `.click()` — разница между двумя вызовами
для ОДНОГО клика меньше миллисекунды, вызовы происходят в одном и том же
синхронном тике.

**Фактическая причина (найдена сверх исходных 4 пунктов, но полностью в
рамках диагностики п.2):** `<input type="file">` физически находится
ВНУТРИ `<div onClick={...}>` как DOM-потомок. Когда `fileRef.current.click()`
вызывается, браузер синтезирует **новое** нативное 'click'-событие на
`<input>`, которое **всплывает вверх по DOM и проходит через тот же самый
`<div>`** — React перехватывает это всплытие через свою систему делегирования
событий и **повторно вызывает `onClick` того же `<div>`**, что приводит к
**повторному, вложенному, синхронному вызову `.click()` на том же `<input
type="file">`** изнутри стека вызовов первого клика. Подтверждено точной
инспекцией события: первое событие — `target: SPAN, isTrusted: true`
(настоящий клик пользователя), второе — `target: INPUT, isTrusted: false`
(синтетическое, порождённое самим `fileRef.current.click()`). Двойной,
вложенный вызов `.click()` на файловом инпуте — известный класс проблем в
разных браузерах: одни (в т.ч. Chromium в этой тестовой среде) молча
игнорируют вложенный повторный вызов и показывают диалог один раз, другие
могут интерпретировать второй вызов (без полноценной, ещё не «использованной»
активации пользовательского жеста, т.к. активация уже была использована
первым вызовом в этом же тике) как невалидный и **не показать диалог
вообще** — то самое поведение, которое описал пользователь: «клик — и
видимо ничего не происходит».

## Фикс (R13)

Причина — не п.3 (Dialog), а результат диагностики п.2 (реальный дубль-вызов
обработчика из-за всплытия события от вложенного `<input>`). Добавлен
`onClick={(e) => e.stopPropagation()}` непосредственно на
`<input type="file">` в `UploadZone.tsx` — останавливает всплытие
синтетического click-события инпута до того, как оно достигнет `onClick`
родительского `<div>`, устраняя повторный вызов.

```diff
 <input
   ref={fileRef}
   type="file"
   accept=".xlsx,.xls,.pdf"
   className="hidden"
   onChange={handleFileChange}
+  onClick={(e) => e.stopPropagation()}
 />
```

Проверено после фикса тем же методом: реальный React `onClick` внешнего
`<div>` теперь срабатывает **ровно один раз** на клик (было 2). Diagnostic
console.log-инструментация полностью убрана из обоих файлов после
подтверждения — в финальном коде остался только сам фикс.

Chromium/CDP-путях, но не во всех реальных путях трассировки событий
браузера». Дальнейшая опора на тот же тип проверки была бы четвёртым

процесс патчинга поверх `ref.click()` показал себя ненадёжным даже с
точки зрения элементарной доставки; (б) реентрантность через всплытие была
лишь частично исчерпывающим диагнозом (сама конструкция `div onClick → 
ref.click()` остаётся источником этого целого класса багов, чинить её
патчем — значит оставлять почву для итерации 5); в `UploadZone.tsx`
применена замена, предписанная R2:

- убран `onClick={() => fileRef.current?.click()}` с внешнего контейнера
  и сам вызов `fileRef.current?.click()`;
- `<input type="file">` получил явный `id="estimate-upload-input"`;
- внешний `<div>` заменён на `<label htmlFor="estimate-upload-input">`,
  оборачивающий **всю дропзону целиком** (весь видимый блок — иконку-текст
  подсказки, «Выберите файл», строку расширений), чтобы сохранить прежнюю
  кликабельную площадь (кликабельна была вся зона, не только слово
  «Выберите файл») — добавлен класс `block`, чтобы `<label>` (инлайновый по
  умолчанию) занимал всю ширину и высоту, как раньше занимал `<div>`;
- `onDragOver`/`onDragLeave`/`onDrop` перенесены с `div` на `label` без
  изменений — drag-and-drop логика (`handleDrop`) не тронута;
- `ref={fileRef}` на `<input>` оставлен как есть (не используется для
  открытия диалога, только зарезервирован под будущую программную очистку
  значения — фактически нигде в файле больше не читается, но раздел 3 прямо
  разрешал его не трогать);
- `stopPropagation` с `<input>` убран за ненадобностью — при триггере через
  `<label>` браузер сам открывает диалог одним нативным кликом на `<input>`,
  без синтетического `.click()` и без повторного всплытия события через
  внешний контейнер: весь класс проблемы «реентрантный клик» структурно
  исключён, а не подавлен ещё одним условием.

Визуально ничего не изменилось: тот же текст, тот же синий подчёркнутый
стиль слова «Выберите файл», тот же размер и форма кликабельной области.

### Итог по разделу 6 (финальная сверка)

Фикс из R1 (`stopPropagation`, итерация 3) **был на месте физически, но
никогда не был закоммичен** — это первый, самостоятельно достаточный факт,
который мог полностью объяснить, почему он не дошёл до пользователя. При
этом реентрантность через всплытие клика — диагноз итерации 3 — подтверждён
как **частично верный, но недостаточный как терминальное решение**: даже
если бы фикс был закоммичен и доставлен, он оставался патчем поверх
конструкции `div onClick → ref.click()`, которая сама по себе — источник
целого класса похожих сбоев (реальный браузер конкретного пользователя мог
воспроизводить их иначе, чем Chromium под Playwright, и ни один из
доступных здесь инструментов не может формально исключить такую разницу).
Поэтому эта итерация не ограничилась повторной проверкой того же патча, а
выполнила R2 — замену механизма на `<label htmlFor>`, что переводит открытие
диалога выбора файла на нативное поведение браузера (HTML label→input
association), не завязанное ни на один JS-обработчик клика и ни на один
`.click()`-вызов. Это исключает данный класс проблем структурно, а не
логически (нет кода, который можно было бы обойти повторным всплытием
события).

## Ограничение проверки (честно, не как гипотеза, а как факт об инструменте)

Playwright в этой среде использует бандл Chromium, управляемый через CDP —
не выделенный канал `chrome` (реальный бинарник Google Chrome) и не ручной
клик человека. Automated-проверка (до и после R2) технически не может
доказать, что диалог откроется у живого пользователя в его личном Chrome —
она может только подтвердить или опровергнуть конкретный класс причин.
Именно поэтому в этой итерации решение об R2 принято не по признаку «тест
покраснел», а по совокупности двух независимых фактов: фикс никогда не был
закоммичен (D1) и лежащая в основе конструкция `ref.click()` остаётся
структурно хрупкой независимо от исхода теста.

**Финальный статус итерации:** Готово с отклонением — не удалось
поставить диагноз со 100% строгостью через доступный инструмент
(Playwright/Chromium не воспроизводит разницу между окружением реального
пользователя и тестовым), поэтому решение об R2 принято на основании
косвенных, но self-sufficient фактов (незакоммиченный фикс + структурная
хрупкость `ref.click()`), а не через прямое воспроизведение отказа. Функционально
все пункты DoD (D1, D3, D4) подтверждены; D2 не применим, т.к. фикс уже
существовал в рабочем дереве.

## Допущения

**1. Проп `docContext` — только `{ project: string }`, без клиента/типа документа.**
R6 буквально требует передавать контекст `{ project: string }` — этого
достаточно, чтобы найти смету по ключу в `SMETA_MOCKS` (ключ — сам проект,
как и в `CreateClientDocModal.PROJECTS`). Расширять контекст сущностями, не
требуемыми ни R6, ни разделом «Что не включать» (там прямо исключён
реальный парсинг файла), не стал.

**2. Раздел 1 при `smetaData` не сохраняет кликабельное замечание "rim".**
Формат размещения (мultiple placements) в оригинале был отдельным
маркированным замечанием в теле документа. Когда передана реальная смета,
раздел полностью заменяется перечислением услуг (буквально по R5: «раздел
перечисляет услуги»), поэтому маркер `rim` в теле документа для этого случая
не рендерится — но само замечание "rim" остаётся в панели «Замечания» и
резолвится независимо (принятие в панели не зависит от наличия маркера в
теле — проверено в браузере, D6 подтверждён). Аналогично для «2. СРОКИ» —
при `smetaData` показывается просто текст периода без интерактивного
маркера "period".

**3. IssueCard.tsx переведён, хотя не назван явно ни в одном R.**
Раздел «Что не включать» ТЗ содержит явную оговорку: «Изменение логики
замечаний (IssueCard) — только перевод текстов», что читается как разрешение
переводить текстовые константы компонента (кнопки «Accept»/«Edit»/…) без
изменения его логики — сделано только это, пропсы/поведение не менялись.

**4. Число «394 000» форматируется собственной функцией, не `Intl`/`toLocaleString`.**
`toLocaleString('ru-RU')` в части окружений возвращает неразрывный пробел
(` `) как разделитель разрядов, который мог бы не совпасть при поиске
текста «394 000» с обычным пробелом в Playwright-проверке (D11). Используется
`replace(/\B(?=(\d{3})+(?!\d))/g, " ")` с обычным пробелом — гарантированно
совпадает с ожидаемой строкой.
