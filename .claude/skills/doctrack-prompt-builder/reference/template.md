# Часть A — как заполнять

## Шаг 0 — выбор пути: полный или лёгкий

Перед заполнением реши, какой путь нужен. Критерий лёгкого пути — все три
условия одновременно:
- задача трогает один файл (или один компонент + его прямой родитель);
- на странице нет одновременного Kanban+Table и задача его не создаёт;
- задача не добавляет новый источник данных, новую роль/уровень доступа.

Все три выполнены → **лёгкий путь**: заполняются только разделы 1, 2a
(ссылкой, без правки), 3 (без 3a), 4, 5, 7. Разделы 3a и 6-таблица не
заполняются, статус в конце — одна строка "Готово" / "Готово с
отклонением: [что]" / "Заблокировано: [что]", без файла-отчёта.

Хотя бы одно условие не выполнено → **полный путь**: все разделы Части B,
включая 3a и таблицу в `IMPLEMENTATION_REPORT.md`.

## Шаг 1 — заполнение разделов

1. Раздел 1 (Заголовок) и 2b (локальная зона) — по задаче, см. ниже как
   заполнять 2b. Раздел 2a не редактируется, вставляется ссылкой как есть.
2. Раздел 3 (R). Неоднозначность в формулировке требования уточняется у
   пользователя здесь, не позже.
3. Раздел 3a (только полный путь) — трассировка R → D.
4. Раздел 4 (D) по каждому R.
5. Раздел 5 (шаги Playwright-проверки).
6. Раздел 7 — копируется дословно из `section7_canonical.txt`, не
   переформулируется.
7. «Что НЕ включать» — заполняется всегда, короткий список или
   "дополнительных ограничений нет".

## Как заполнять раздел 2b (локальная зона "не трогать")

Раздел 2b не придумывается — он выводится механически:
1. Выпиши все символы (компоненты, хуки, утилиты, типы), которые
   упоминаются в разделе 3 как редактируемые или используемые.
2. По каждому символу — grep по репозиторию: кто его импортирует.
3. Файлы-импортёры, которые эта задача не редактирует напрямую — это и
   есть раздел 2b, поимённо.
4. Если среди них есть общий источник данных/фильтрации (см. правило
   Kanban+Table в 2a) — отметь его отдельной строкой с пометкой "особо
   чувствительно: единый источник данных".

Если grep не дал ни одного стороннего импортёра — раздел 2b состоит из
одной строки "сторонних импортёров не найдено".

## Самопроверка перед показом пользователю

Выполняется после заполнения всех разделов Части B, перед показом. Не
попадает в `prompts/NN-название.md`. При провале пункта — правь только
указанную ячейку "Где править" и перепроверяй только указанную в "Что
перепроверить" — не весь процесс заново, кроме строки про шаг 0.

| Пункт | Где править | Что перепроверить |
|---|---|---|
| R неполный (нет что/как/на выходе) | этот R в разделе 3 | его D, его строка 3a |
| D не бинарен / без слоя | этот D в разделе 4 | шаг раздела 5 |
| ссылка на код не проверена только что | эта ссылка в 2b или 3 | — |
| пересекающийся UI-текст без id | этот R (добавить data-testid) | шаг раздела 5 |
| раздел пустой или отсутствует | добавить раздел | — |
| выбран не тот путь (лёгкий/полный) | шаг 0 | все разделы заново |

Все пункты выполнены → показывай Часть B пользователю, компромисс (если
был) — одной строкой.

## 5) Проверка через Playwright (MCP, реальный браузер)
Цикл на каждый D: действие → снапшот/скриншот → сравнение с величиной из
D → pass/fail. Fail критический — чинить сразу и повторить шаг; fail
некритический — пометить и идти дальше. Последний шаг — консоль на 0
ошибок сразу после последнего релевантного действия.

Проверка идёт через инструменты Playwright MCP по живому dev-серверу, не
через testing-library/jest/vitest (их в проекте нет).
