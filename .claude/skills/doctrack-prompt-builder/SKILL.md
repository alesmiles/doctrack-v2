При сохранении файла в `prompts/` это запускается автоматически через хук
(см. `.claude/settings.json`), результат приходит в контекст без отдельного
вызова — но если по какой-то причине хук не сработал, прогони вручную перед
шагом 2.2. Линт ловит только механические проблемы (пропущенные/пустые разделы,
запрещённые формулировки без ссылки на файл, расхождение раздела 7 с эталоном,
элементы раздела 6, трассировку, покрытие, ссылки на несуществующий код,
конфликт "или"/скобочных вариантов в разделе 4, тестовые матчеры в разделе 5).
Прохождение линта не равно готовности — переходи к шагу 2.2 в любом случае.

### Шаг 2.2 — смысловая сверка (то, что линт не может проверить)

Зафиксируй по каждому пункту: выполнено / не выполнено, с обоснованием.

- [ ] Каждое требование раздела 3 — проверяемая величина (px/hex/текст/имя
      символа), а не описательное прилагательное или решение, отданное на
      усмотрение исполнителя.
- [ ] Каждый пункт DoD раздела 4 — да/нет без интерпретации, без внутренней
      альтернативы ("или", перечисление через "/").
- [ ] Ни один D-id не проверяет два независимых факта одновременно.
- [ ] Ни одно требование не содержит ветку, которая не наступает ни при одном
      реальном входном состоянии — если ветка описана, покажи конкретный
      сценарий, в котором она сработает.
- [ ] Каждая ссылка на существующий код проверена в файле только что, а не
      взята по памяти о прошлой сессии.
- [ ] Незнакомый с задачей разработчик выполнит промпт без единого уточняющего
      вопроса — включая случаи, когда два требования потенциально пересекаются
      на одном и том же UI-элементе (напр. одинаковый текст в разных разделах
      интерфейса) — такие пересечения адресуются явно, через конкретный
      идентификатор/атрибут, а не через текст, который может встретиться
      дважды.

Если хоть один пункт "не выполнено" — вернись в фазу 1/1a, исправь, повтори
2.1 и 2.2 целиком заново.

### Шаг 2.3 — финальный вывод

Показывай промпт пользователю только после чистого линта и всех пунктов 2.2
"выполнено". Если пришлось пойти на компромисс — одной строкой скажи об этом
пользователю, не молчи.

## Фаза 3 — проверка присланного промпта

Тот же процесс 2.1–2.2 применяется к присланному тексту напрямую. Результат —
список конкретных несоответствий, не переписанный промпт.

## Файлы этого skill

- `reference/skeleton.md` — скелет для новой генерации (копируется, не читается
  как описание).
- `reference/skeleton_delta.md` — скелет для доработки после итерации.
- `reference/section7_canonical.txt` — эталонный текст раздела 7.
- `reference/banned_phrases.txt` — запрещённые оценочные формулировки.
- `scripts/lint_prompt.py` — механический линтер.

## Важно

Изменение самих скелетов — отдельное действие, не выполняется молча внутри
задачи по генерации промпта.
